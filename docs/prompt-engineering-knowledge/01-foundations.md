# 01. 基础模型：提示词工程到底在工程化什么

提示词工程不是“写一句更聪明的话”，而是把任务、上下文、工具、约束、输出、评估和迭代做成可维护系统。一个成熟提示词应该能被他人理解、复用、测试、追踪和升级。

## 核心定义

提示词工程关注三件事：

1. **任务表达**：让模型知道要解决什么问题、为谁解决、成功标准是什么。
2. **上下文组织**：让模型获得足够但不过量的事实、样例、工具说明和边界。
3. **行为控制**：用结构、输出契约、评估标准、拒绝策略和工具调用规则约束输出。

在当前项目中，提示词工程进一步被拆成资产：Prompt、Skill、MCP、SDK、Workflow、Reference、Agent、Tool、Template、Evaluator、Dataset、Policy、Memory、Connector、Parser、Benchmark。

## 一个可工程化提示词的最小闭环

一个可沉淀的提示词至少包含：

- **目标**：这段提示词要稳定完成什么任务。
- **输入契约**：用户、系统、文件、工具、知识库分别提供什么。
- **处理过程**：需要分析、拆解、调用工具、比较、反思还是直接生成。
- **输出契约**：格式、字段、长度、语言、风格、可验证证据。
- **约束边界**：不能做什么，遇到缺失信息怎么办。
- **评价标准**：怎样判断结果好、坏、可接受、需要重试。
- **迭代方式**：失败后如何定位是上下文、指令、工具、模型还是样例的问题。

## 从自然语言到资产库的转变

传统提示词常见问题：

- 写在聊天记录里，无法复用。
- 依赖个人记忆，无法交接。
- 没有版本和评估，模型升级后难以回归。
- 把工具、资料、规则、样例混在正文里，后续难以维护。
- 没有明确成功标准，优化只凭感觉。

资产库做法：

- 把稳定任务写成 Prompt 或 Template。
- 把稳定工作方法写成 Skill。
- 把外部能力写成 Tool、MCP、SDK 或 Connector。
- 把领域资料写成 Reference、Memory、Dataset。
- 把安全和业务边界写成 Policy。
- 把质量标准写成 Evaluator 和 Benchmark。
- 在优化时按需求搜索、推荐、人工确认后注入。

## 上下文优先级

模型通常需要处理多层上下文。建议在提示词和资产中明确优先级：

1. 系统安全和不可违反约束。
2. 当前用户任务目标。
3. 用户显式选择的资产。
4. 当前输入文件和材料。
5. 推荐但未确认的资产。
6. 通用最佳实践和模型默认能力。

这样可以减少资产冲突和过度注入。

## 结构化不是目的，稳定性才是目的

提示词结构化常见形式包括：

- 角色、背景、任务、输入、步骤、约束、输出格式、验收标准。
- XML 标签：`<context>`、`<task>`、`<examples>`、`<constraints>`、`<output_format>`。
- JSON Schema 或表格字段。
- 分阶段工作流：澄清、计划、执行、检查、交付。

结构的价值不在于好看，而在于：

- 降低模型误解。
- 便于局部替换。
- 便于自动注入资产。
- 便于评估和回归。
- 便于跨模型迁移。

## 什么时候需要工程化

适合进入项目库的提示词：

- 高频重复任务。
- 对输出质量、格式或安全边界有稳定要求。
- 需要工具、SDK、MCP 或资料配合。
- 需要多轮迭代或多人协作。
- 错误成本高，需要评估和回归。

不一定需要沉淀的提示词：

- 一次性闲聊。
- 低风险、低重复、低约束任务。
- 目标还没有稳定下来，仍处于探索阶段。

## 提示词工程师的工作流

1. **澄清任务**：谁使用、输入是什么、交付是什么、风险是什么。
2. **选择资产**：从库中挑 Prompt、Skill、Tool、Reference、Policy、Evaluator。
3. **组装提示词**：按目标把资产融合成单次可执行指令。
4. **运行和观察**：记录输出、错误、用户修改点。
5. **评估**：用 rubric、样例集、人工复核或自动测试判断。
6. **沉淀**：更新资产、版本、示例和失败案例。
7. **回归**：模型升级、工具变化、资料更新后重新验证。

## 最常见的失败类型

- 目标模糊：模型不知道“好”的标准。
- 上下文缺失：模型只能编造或泛化。
- 上下文过载：关键信息被埋没，长文本互相冲突。
- 输出格式不稳定：没有 schema、示例或强约束。
- 工具边界不清：模型声称调用了并未调用的工具。
- 样例污染：few-shot 示例质量低或与当前任务不匹配。
- 评估缺失：看起来变好了，但没有证据。
- 安全缺失：提示注入、数据泄露、越权工具调用。

## 与当前项目的对应关系

当前项目的优化器应该把用户输入看作“待工程化需求”，而不是只做润色：

- 用户输入决定任务。
- 场景和风格决定默认表达。
- 优化方向决定增强策略。
- 用户确认资产决定可引用上下文。
- 推荐资产作为弱参考，不能覆盖用户显式选择。
- 最终 `highlights` 应说明融合了哪些资产和方向。
