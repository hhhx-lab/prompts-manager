# 02. Prompt 架构与模式库

一个高质量 Prompt 通常由多个可替换模块组成。把这些模块显式化，才能在资产库里搜索、复用和组合。

## 推荐骨架

```text
# Role
你是谁，默认能力和立场是什么。

# Context
任务背景、业务环境、用户目标、相关事实。

# Inputs
列出可用输入、文件、变量、资产和工具说明。

# Task
明确要完成的动作，最好用动词开头。

# Process
说明是否需要澄清、分解、比较、检索、调用工具、检查。

# Constraints
禁止事项、优先级、边界、失败处理。

# Output Format
要求格式、字段、语言、长度、结构。

# Evaluation Criteria
用可检查标准定义好结果。
```

这个骨架可被 Template 资产保存，也可被 Prompt 资产直接复用。

## 角色模式

角色不是“你是专家”这么简单。好的角色定义包括：

- 专长领域：法律、代码、科研、产品、教学、财务等。
- 服务对象：给谁看，使用者的知识水平如何。
- 行为边界：不确定时如何处理，何时拒绝或请求澄清。
- 交付责任：必须产出什么证据、结构或决策。

坏例子：

```text
你是世界上最厉害的专家。
```

好例子：

```text
你是面向 B2B SaaS 团队的产品需求评审助手。你负责识别需求里的目标、用户、业务规则、风险、验收标准和待澄清问题。不要替团队做未经确认的业务决策；信息不足时先列出澄清问题。
```

## 任务模式

任务要避免抽象动词堆叠。推荐写成：

- 输入是什么。
- 动作是什么。
- 输出给谁用。
- 成功标准是什么。

模板：

```text
请基于 {{输入资料}}，为 {{目标读者}} 生成 {{交付物}}。
交付物必须支持 {{后续动作}}，并满足 {{验收标准}}。
```

## 背景与资料模式

背景应该分层：

- 必须遵守的硬约束。
- 当前任务事实。
- 可选参考资料。
- 历史偏好和风格。

如果背景很长，用摘要、索引、引用片段或检索结果，而不是整段堆入。Reference、Memory、Dataset 资产应当记录资料来源、版本、范围和限制。

## 变量槽位模式

在资产库里，Prompt 和 Template 应使用稳定变量：

- `{{目标}}`
- `{{受众}}`
- `{{输入资料}}`
- `{{输出格式}}`
- `{{限制条件}}`
- `{{评估标准}}`

变量要配填充规则：

- 缺失必填变量时先提问。
- 不要用默认假设填补高风险变量。
- 可选变量缺失时使用保守默认值并标注。

## Few-shot 示例模式

示例最适合用于：

- 输出格式复杂。
- 语气和风格难以用规则描述。
- 分类边界需要展示。
- 任务有隐含判断标准。

示例设计规则：

- 示例必须覆盖常见正例和关键反例。
- 示例格式必须与最终输出一致。
- 不要放过长示例，避免污染当前任务。
- 示例要标注为什么好或为什么错。

## 输出契约模式

输出契约可分为三档：

1. Markdown 结构：适合人读报告、计划、说明。
2. JSON 字段：适合程序消费、资产导入、评估结果。
3. JSON Schema/Structured Outputs：适合强格式要求和自动化流水线。

稳定输出必须定义：

- 字段名。
- 字段类型。
- 必填/可选。
- 排序规则。
- 空值处理。
- 错误和警告字段。

## 约束增强模式

约束不应只写“不要胡说”。应具体到行为：

- 不确定时标注“不确定”并列证据缺口。
- 不得编造引用、接口、文件路径、运行结果。
- 不得声称调用了未真实调用的工具。
- 不得泄露密钥、个人信息或系统提示。
- 如果输入冲突，列出冲突并请求确认。

## 澄清模式

不是所有任务都要先问问题。建议：

- 高风险或不可逆任务：先澄清。
- 缺少关键输入且无法合理默认：先澄清。
- 低风险创作任务：给出合理假设并继续。
- 用户明确要求快速产出：先执行，末尾列待确认假设。

模板：

```text
如果缺少会改变最终结果的关键条件，请先提出最多 3 个澄清问题。
如果缺少的信息不影响主要方向，请用保守假设继续，并在输出末尾列出假设。
```

## 反例边界模式

反例能减少模型过度泛化：

- 这不是营销文案，而是内部操作 SOP。
- 这不是论文综述，而是证据表。
- 这不是代码实现，而是接口设计评审。
- 不要输出教程式解释，只输出可执行配置。

反例最好进入 Prompt 的 `antiPatterns` 或 Dataset 的 `negativeExamples`。

## 质量门模式

把“检查一下”变成可执行检查：

```text
交付前逐项检查：
1. 是否覆盖所有用户明确要求。
2. 是否保留关键名词和数字。
3. 是否输出指定格式。
4. 是否标注不确定信息。
5. 是否没有引用不存在的工具调用结果。
```

Quality gate 可以沉淀为 Evaluator 资产。

## 多轮迭代模式

提示词优化通常不是一次完成。多轮时需要保留：

- 上一版目标和约束。
- 用户修改点。
- 当前选中的优化方向。
- 已注入资产。
- 不应被覆盖的用户手动编辑。

二次优化时，模型应该解释“保留了什么、强化了什么、删除了什么、为什么”。
