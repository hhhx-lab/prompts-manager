# 06. 评估、迭代与回归

没有评估的提示词优化只是改写。工程化提示词必须能回答：“它真的更好吗，证据是什么？”

## 先定义成功

评估前先明确：

- 任务目标。
- 用户群体。
- 必须满足的硬性约束。
- 可接受的失败率。
- 人工复核标准。
- 输出应该如何被下游使用。

Anthropic 官方评估方法强调先定义成功标准，再设计测试。OpenAI Evals 和 GitHub Models 等工具也体现了同样思路：把 prompt 结果放入可重复试验。

## 评估维度

常用维度：

- 准确性：事实、计算、逻辑是否正确。
- 完整性：是否覆盖所有需求。
- 可执行性：输出能否直接用于下一步。
- 格式合规：是否满足 schema 或模板。
- 稳定性：多次运行是否一致。
- 安全性：是否违反 Policy 或泄露数据。
- 可引用性：是否给出来源和证据。
- 成本效率：是否过长、过慢、过度调用工具。

Evaluator 资产应把这些维度写成 scoringRubric。

## Rubric 设计

好的评分标准应具体、可判定：

```text
准确性 0-5
5: 所有事实均由输入或引用来源支持，无编造。
3: 主结论正确，但有轻微未证实表述。
1: 存在影响决策的事实错误。
0: 主要结论错误或无法使用。
```

避免：

- “内容要好”。
- “逻辑清晰”但不定义清晰标准。
- 所有维度都主观打分，无失败样例。

## Dataset 与 Benchmark

Dataset 保存样例和标签；Benchmark 保存任务、输入、期望输出和指标。

建议拆分：

- smoke：少量关键样例，快速检查。
- regression：覆盖历史失败和核心业务。
- stress：长上下文、边界条件、恶意输入。
- golden：人工确认的高质量期望输出。

## A/B 测试

A/B 测试适合比较：

- 两版 Prompt。
- 不同输出格式。
- 是否注入某个资产。
- 不同优化方向。
- 不同模型。

比较时要固定：

- 输入。
- 评估器。
- 模型参数。
- 工具可用性。
- 样例集。

否则不能判断差异来自哪里。

## 失败分析

失败要归因到具体层：

- Prompt 指令不清。
- 输入缺少关键信息。
- Reference 过期或冲突。
- Tool/MCP schema 不完整。
- 输出格式约束不足。
- 示例误导。
- Policy 未覆盖风险。
- 模型能力不适合任务。

失败记录应该反哺资产：

- 加入 Dataset negativeExamples。
- 更新 Evaluator failureCases。
- 更新 Prompt antiPatterns。
- 更新 Policy triggers。
- 更新 Workflow failureHandling。

## 自动优化流程

一个可落地的闭环：

1. 收集真实任务和失败案例。
2. 标注期望输出和失败类型。
3. 生成候选 Prompt。
4. 在 Benchmark 上运行。
5. 用 Evaluator 打分。
6. 人工复核高风险项。
7. 保存最佳版本和变更说明。
8. 记录适用场景和不适用范围。

## 回归触发条件

以下变化必须回归：

- 模型版本变化。
- SDK/MCP 工具 schema 变化。
- 重要 Reference 更新。
- Policy 或业务规则更新。
- Prompt 结构改动。
- 示例集改动。
- 用户报告关键失败。

## 当前项目中的评估建议

当前项目已有历史版本、版本对比和 A/B 测试。下一步可以把资产库接入评估：

- 优化结果 highlights 必须说明使用了哪些资产。
- 每个资产可以关联 Evaluator 或 Benchmark。
- 资产导入后可跑基础格式检查。
- Prompt 优化可以保存 selectedAssetIds 和 directions。
- 对高频任务建立 Dataset，作为 Prompt 回归集。
