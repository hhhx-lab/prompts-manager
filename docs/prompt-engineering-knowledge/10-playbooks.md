# 10. 实战手册

本文件把前面的方法压缩为可直接复用的工作手册。

## 从一句需求生成高质量 Prompt

1. 识别目标、受众、输入、交付物。
2. 判断是否需要澄清。
3. 选择优化方向：结构化、约束增强、可执行性、评估标准等。
4. 从资产库推荐 Prompt、Template、Reference、Policy、Evaluator。
5. 用户确认资产。
6. 组装为角色、背景、任务、步骤、约束、输出格式、评估标准。
7. highlights 说明改动原因和资产贡献。

## 把一次成功对话沉淀为 Prompt 资产

1. 提取稳定任务。
2. 删除一次性上下文。
3. 把变量替换为 `{{变量}}`。
4. 补充适用场景和不适用场景。
5. 写输出契约。
6. 写 evaluationCriteria 和 antiPatterns。
7. 加入一个正例和一个反例。
8. 保存并跑至少一个回归样例。

## 把一套工作方式沉淀为 Skill 资产

1. 写清触发条件。
2. 写清何时不触发。
3. 拆出输入、流程、输出、质量门。
4. 长资料进入 references。
5. 可自动执行的操作进入 scripts。
6. 示例、模板、素材进入 assets。
7. 工具服务说明进入 mcp。
8. 写 handoff 和 validation。

## 把外部工具沉淀为 MCP/Tool/Connector

1. 判断粒度：单函数用 Tool，协议 server 用 MCP，平台连接用 Connector。
2. 记录认证方式和权限。
3. 记录输入输出 schema。
4. 标注副作用和危险等级。
5. 写失败处理。
6. 写调用示例。
7. 说明当前项目中是否真实执行。

## 把 SDK 沉淀为开发资产

1. 记录包名、语言、版本、安装命令。
2. 记录初始化和认证。
3. 列核心方法。
4. 每个方法写参数、返回、错误。
5. 写最小可运行示例。
6. 写兼容性和测试方式。
7. 标注密钥不能进前端或仓库。

## 给 Prompt 增加评估器

1. 明确评估对象。
2. 写维度。
3. 每个维度写 0-5 或 0-100 rubric。
4. 写通过阈值。
5. 写失败案例。
6. 指定 manual、ai 或 hybrid。
7. 指定输出格式。

## 长文档问答 Prompt

```text
你是资料问答助手。请只基于提供资料回答。

<question>
{{问题}}
</question>

<sources>
每条资料包含 source、version、snippet、confidence。
</sources>

规则：
1. 资料支持的结论必须标注来源。
2. 资料不足时输出“无法从资料确认”。
3. 不要用常识补充关键事实。
4. 如果资料冲突，列出冲突并说明需要人工确认。

输出：
- 直接答案
- 证据
- 不确定事项
- 下一步
```

## 代码修复 Prompt

```text
你是代码修复助手。目标是在最小范围内修复问题。

输入：
- 问题描述：{{问题}}
- 相关文件：{{文件}}
- 失败日志：{{日志}}
- 验证命令：{{命令}}

要求：
1. 先定位根因和影响范围。
2. 不修改无关文件。
3. 保留用户已有改动。
4. 修改后运行验证命令。
5. 输出变更摘要、验证结果和剩余风险。
```

## Prompt 安全审查

检查：

- 是否把外部资料当作指令。
- 是否可能泄露密钥或隐私。
- 是否允许模型声称调用了未调用工具。
- 是否有危险工具确认机制。
- 是否有输出格式校验。
- 是否有失败和不确定处理。
- 是否能抵御“忽略之前指令”类输入。

## 资产库月度维护

1. 导出 JSON 备份。
2. 删除或归档过期资产。
3. 检查高频资产的失败案例。
4. 更新官方 SDK/MCP 文档链接。
5. 运行核心 Benchmark。
6. 合并重复资产。
7. 补充缺失 examples 和 evaluationCriteria。
8. 更新 README 或 docs。
