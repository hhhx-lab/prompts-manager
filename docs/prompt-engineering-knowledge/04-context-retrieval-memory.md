# 04. 上下文、检索、长文本与记忆

上下文质量往往比提示词措辞更重要。模型不知道的信息，不能靠“请认真一点”补出来。

## 上下文的四类来源

1. **用户输入**：当前任务、目标、偏好、限制。
2. **项目资产**：Prompt、Skill、Tool、Reference、Policy、Memory 等。
3. **外部资料**：文件、网页、知识库、API、数据库。
4. **运行状态**：历史版本、测试结果、工具输出、失败记录。

提示词应明确每类上下文的可信度和优先级。

## 长上下文压缩

长上下文不是越多越好。常见问题：

- 关键需求被淹没。
- 旧信息覆盖新信息。
- 多个资料互相矛盾。
- 模型只关注靠近末尾的内容。

推荐压缩策略：

- 先生成资料索引，再提取与任务相关片段。
- 保留标题、来源、版本和关键事实。
- 对表格、代码、日志保留结构化摘要。
- 对低相关内容只保留链接或一句摘要。
- 对冲突内容显式列出冲突点。

## RAG 提示词结构

检索增强任务建议包含：

```text
# Question
{{用户问题}}

# Retrieved Context
每条资料包含 source、date、snippet、confidence。

# Instructions
只使用资料能支持的事实。
资料不足时说明缺口。
引用结论时标注来源。

# Output
答案、证据、无法确认事项、下一步。
```

Reference 资产应提供 source、version、scope、keyFacts、citationRules、limitations、freshness。

## 记忆的边界

Memory 资产不等于永久真理。它适合保存：

- 用户稳定偏好。
- 项目约定。
- 团队术语。
- 长期事实。
- 工程流程。

它不适合保存：

- 短期临时输入。
- 未验证事实。
- 密钥、令牌、隐私数据。
- 会频繁变化的价格、排期、人员状态。

每条 Memory 都应该有 scope、confidence、updatedAtText、invalidationRules。

## 上下文冲突处理

当资料冲突时，Prompt 应指定处理方式：

1. 最新用户明确指令优先。
2. 安全和合规 Policy 优先于任务便利性。
3. 有来源和版本的 Reference 优先于无来源描述。
4. 文件中的明确数据优先于模型常识。
5. 如果冲突会改变结果，先请求确认或列出分支方案。

## 文档类上下文

Markdown、Word、PDF、Excel 等文件导入时，应尽量转成可引用结构：

- 标题和章节。
- 表格摘要。
- 关键实体。
- 日期和版本。
- 原文片段引用。
- 未解析或低置信内容。

Parser 资产负责定义提取字段、清洗规则、输出 schema 和失败处理。

## 示例集上下文

Dataset 资产保存 few-shot 示例、反例和标签。高质量示例集应该：

- 覆盖主场景。
- 覆盖边界和失败情况。
- 每条样例有标签。
- 期望输出可评估。
- 不混入过时或错误示例。

## 上下文注入预算

资产推荐时不要盲目注入全部资产。建议：

- 默认推荐前 5 个。
- 用户最多确认 8 个。
- 对每个资产只注入摘要、能力、输入输出、约束和必要正文。
- 长正文使用节选或摘要。
- `highlights` 说明具体使用了哪些资产。

## 上下文可信度标注

推荐字段：

- `source`: 来源。
- `freshness`: 更新时间或时效性。
- `confidence`: 高/中/低。
- `scope`: 适用范围。
- `limitations`: 不适用范围。

这能防止 Reference 和 Memory 被模型过度泛化。
